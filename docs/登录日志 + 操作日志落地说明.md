# 登录日志 + 操作日志落地说明

## 1. 改造目标

本次改造将认证中心日志体系拆分为两条链路：

- 登录日志：落 MySQL，服务于后台审计与稳定分页查询
- 操作日志：主存储落 Elasticsearch，服务于高并发写入、检索与审计查询

核心目标不是“把日志记下来”，而是避免日志系统在高并发下拖垮主业务。

## 2. 当前现状分析

### 2.1 模块结构与公共基础设施

- `sso-auth`：认证、会员、后台管理接口，承载本次登录日志查询与操作日志查询接口
- `sso-log`：本次新增的操作日志统一能力模块，负责注解、AOP、脱敏摘要、异步队列、ES 批量写入、查询封装
- `sso-web`：Web 层基础设施，已提供统一异常处理、限流、防重复提交、请求上下文工具
- `sso-core`：核心配置、Feign、虚拟线程配置、通用枚举/序列化/分页模型
- `sso-gateway`：网关入口，本次补齐 traceId/requestId 透传
- `sso-mybatis`：MyBatis Plus 基础设施

### 2.2 Sa-Token 触发点

后台管理员：

- 登录：`AuthSysController.login` -> `AuthServiceImpl.login` -> `StpUtil.login(...)`
- 登出：`AuthSysController.logout` -> `StpUtil.logout()`

会员端：

- 登录：`AuthMemberController.memberLogin` -> `MemberAuthServiceImpl.login` -> `StpMemberUtil.login(...)`
- 注册自动登录：`AuthMemberController.register` -> `MemberAuthServiceImpl.register`
- 微信登录：`AuthMemberController.wechatLogin` -> `WechatAuthServiceImpl.loginMemberByWechatCode`
- 会员登出：`AuthMemberController.memberLogout` -> `StpMemberUtil.logout()`

### 2.3 既有日志能力

- V2 已存在 `t_auth_login_log`、`t_auth_operate_log` 表，但没有完整服务、Mapper、Controller 查询实现
- 旧操作日志表位于 MySQL，不适合海量高并发写入，已改为“历史保留表”
- 后台原先没有现成的登录日志/操作日志查询接口

### 2.4 既有基础能力判断

- AOP：已有，且本次新增 `@OperationLog`
- 线程池：已有虚拟线程配置，本次新增独立日志线程池
- Spring 事件机制：已有
- 统一异常处理：已有 `GlobalExceptionHandler`
- traceId/requestId：原来仅网关有 `X-Request-Id`，本次补齐网关 + 服务端 Filter + MDC + Feign 透传
- Jackson 统一配置：已有 Web 层统一配置，本次查询 DTO 追加时间反序列化

### 2.5 V2 表结构现状

- `t_auth_login_log`：字段偏简单，仅有 `username/login_ip/browser/os/status/msg/login_time`
- `t_auth_operate_log`：旧设计把请求参数、返回值、错误信息落 MySQL `text` 字段

问题在于：

- 无法承载操作日志高吞吐写入
- 请求/响应字段没有严格长度边界
- 没有异步化和批量化写入设计

### 2.6 缺口分析

- 缺少登录日志完整落库链路
- 缺少统一操作日志注解/AOP/手动 API
- 缺少 ES 客户端封装、索引模板、批量写入、失败降级
- 缺少独立日志线程池与有界队列
- 缺少 traceId/requestId 在服务内与 Feign 链路中的贯穿
- 缺少后台查询 API 与深分页风险控制

### 2.7 风险点分析

- 同步写日志会拖慢主流程
- 无边界采集会把大对象、异常栈、响应体直接打爆日志链路
- ES 不可用、Bulk 部分失败、队列打满时如果没有降级，会反向影响主业务
- `ext` 如果放任动态映射，会产生 mapping explosion
- 旧操作日志落 MySQL 模式不利于未来多平台统一接入

### 2.8 改造优先级

1. traceId/requestId 贯穿
2. 登录日志异步落 MySQL
3. 操作日志注解/AOP + 手动 API
4. ES 模板、索引策略、批量写入链路
5. 失败降级、统计指标、后台查询能力
6. 文档与接入规范

## 3. 总体架构

### 3.1 为什么登录日志放 MySQL

- 登录事件量级相对可控
- 后台审计更依赖稳定分页、精确筛选与详情查看
- MySQL 成本低、查询模型简单、便于保留长期审计记录

### 3.2 为什么操作日志不能放 MySQL

- 操作日志量大、写入频繁
- 如果将请求摘要、响应摘要、错误摘要大量写 MySQL，会直接放大主库压力
- 文本检索、模糊查询、多维组合查询不适合关系型主表承担

### 3.3 为什么操作日志适合放 ES

- 天然适合日志检索与多条件过滤
- 支持 keyword/text/date/long 等混合字段模型
- 支持按天滚动、生命周期管理、冷热分层、保留期清理

### 3.4 业务主链路如何解耦

链路如下：

`Controller / Service -> @OperationLog AOP 或手动 API -> 轻量摘要化 -> OperationLogClient -> 有界队列 -> 独立日志线程池 -> Bulk 批量写 ES`

主业务线程不做同步逐条 ES 写入。

## 5. 操作日志 ES 设计与实现

### 5.1 文档模型

固定字段位于 `sso-log/src/main/java/com/zhanglx/sso/log/domain/model/OperationLogDocument.java`

字段分类：

- `keyword`：`logId/appCode/appName/platformCode/platformName/module/feature/operationType/operationName/userId/username/displayName/tenantId/requestMethod/resultStatus/errorCode/exceptionType/traceId/requestId/sourceSystem`
- `text`：`operationDesc`，`requestPath.text`
- `date`：`startTime/endTime/ingestTime`
- `long`：`durationMs`
- `ip`：`clientIp`
- `flattened`：`ext`
- 禁止全文索引：`requestQuery/requestBodySummary/responseSummary/errorMessageSummary/exceptionStackSummary/userAgent`

完整模板：`sso-log/src/main/resources/es/operation-log-index-template.json`

关键控制点：

- 顶层 `dynamic: false`
- `ext` 使用 `flattened`
- `mapping.total_fields.limit: 128`

### 5.2 索引策略

- 命名规则：`operation-log-YYYY-MM-DD`
- 每日滚动
- 模板匹配：`operation-log-*`

建议生命周期：

- Hot：7 天
- Warm：30 天
- Cold：90 天
- Delete：180 天

权衡说明：

- 单索引永久增长会导致段合并、查询和恢复成本持续上升
- 按天滚动会增加索引数量，但查询可以结合时间范围控制扫描面

### 5.3 写入链路

关键实现：

- 注解：`sso-log/src/main/java/com/zhanglx/sso/log/annotation/OperationLog.java`
- 切面：`sso-log/src/main/java/com/zhanglx/sso/log/aop/OperationLogAspect.java`
- 手动 API：`sso-log/src/main/java/com/zhanglx/sso/log/service/OperationLogClient.java`
- 客户端实现：`sso-log/src/main/java/com/zhanglx/sso/log/service/impl/DefaultOperationLogClient.java`
- 队列与批量分发：`sso-log/src/main/java/com/zhanglx/sso/log/infrastructure/queue/OperationLogDispatcher.java`
- ES 客户端：`sso-log/src/main/java/com/zhanglx/sso/log/infrastructure/es/OperationLogElasticsearchClient.java`

双触发提交：

- 满足 `bulkBatchSize` 立即刷写
- 未满批次时，达到 `bulkFlushIntervalMs` 也会刷写

### 5.4 失败与降级

- ES 不可用：记录内部错误日志并有限次重试，重试后仍失败则丢弃
- Bulk 部分失败：仅重试失败子集
- 队列满：根据策略 `DROP_CURRENT` 或 `DROP_OLDEST`
- JSON 序列化失败：在摘要层直接降级为字符串
- 日志异常：绝不反向影响主业务

### 5.5 脱敏与截断

核心实现：`sso-log/src/main/java/com/zhanglx/sso/log/support/OperationLogSummarySanitizer.java`

敏感字段统一脱敏：`password/token/secret/authorization/phone/idCard` 等

长度上限：

- `requestBodySummary`：2048
- `responseSummary`：2048
- `errorMessageSummary`：1024
- `exceptionStackSummary`：4096
- `requestQuery`：1024
- `userAgent`：512

- 精确过滤：`term`

- 普通后台列表先支持 `from + size`
- 超过 `searchMaxWindow` 强制改用 `searchAfterToken`

## 9. 迁移注意事项

- 当前 `application.yaml` 仍保持 `flyway.enabled: false`
- 原因是现有 V1/V2 初始化脚本包含建表/清表逻辑，直接开启 Flyway 存在历史环境误执行风险
- 建议由 DBA 或发布流水线执行 `V3__log_system_upgrade.sql`
