# 前端 XSS 协同接入说明

## 1. 这次为什么需要更新

需要更新，且本轮已经完成更新。
原因是后端 XSS 能力已经从 `sso-web` 独立抽取为 `sso-xss`，同时新增了：

- multipart 文本分片防护
- XSS 审计指标
- 字段级富文本协议清单

如果前端仍按旧说明理解，会出现接入口径和排查路径不一致的问题。

## 2. 总体原则

- 后端为主：后端负责全局拦截、基础清洗、白名单和指标审计
- 前端为辅：前端负责输入约束、危险渲染治理、富文本组件安全配置和展示层转义

不要依赖“后端已经清洗，所以前端可以直接渲染 HTML”。

## 3. 前端现在需要知道的后端事实

### 3.1 后端模块

后端 XSS 能力已经沉淀为独立 starter：`sso-xss`。

### 3.2 当前默认覆盖的输入入口

后端默认已经覆盖：

- Query 参数
- Form 表单参数
- JSON Body
- PathVariable
- 指定请求头
- multipart/form-data 文本字段
- multipart/form-data 文本分片（`@RequestPart String`）
- multipart/form-data JSON 分片（`@RequestPart DTO`）

后端不会清洗：

- `MultipartFile` 二进制文件内容
- 被显式标记为 `NONE` 的字段原文

### 3.3 默认特殊字段

默认放行字段：

- `password`
- `oldPassword`
- `newPassword`
- `confirmPassword`
- `credential`
- `token`
- `accessToken`
- `refreshToken`
- `secret`
- `clientSecret`
- `authorization`

默认搜索字段：

- `searchKey`
- `keyword`
- `query`
- `q`

默认富文本字段：

- 当前项目没有全局放开的富文本字段
- 新增富文本字段前，必须先登记到《XSS富文本字段协议清单》

## 4. 前端必须做的事情

### 4.1 普通文本输入

对于名称、标题、备注、描述、关键字、路径、别名等普通文本：

- 做基础长度校验
- 不要允许整段 HTML 作为普通文本录入
- 不要把普通文本字段当富文本字段处理
- 搜索框不要自己做激进替换，保留用户搜索语义

### 4.2 展示层安全

普通文本展示必须按文本节点渲染：

- Vue：优先 `{{ text }}`
- React：优先普通 JSX 文本节点
- 组件库：优先纯文本插槽 / 文本属性

禁止默认使用：

- `v-html`
- `dangerouslySetInnerHTML`
- `innerHTML` 动态拼接用户输入

### 4.3 URL 与 DOM 拼装

禁止：

- 把用户输入直接拼成 `javascript:` 链接
- 把用户输入直接拼进 HTML 字符串后插入 DOM
- 把用户输入直接写到内联事件属性中

推荐：

- 使用框架路由 API
- 使用文本节点
- 使用组件化渲染

## 5. 富文本场景怎么接

### 5.1 什么时候允许富文本

只有明确的富文本业务场景才允许，例如：

- 公告正文
- 帮助中心正文
- 模板正文
- 富文本消息内容

以下字段默认不是富文本：

- 名称
- 标题
- 备注
- 描述
- 搜索条件
- 标签名
- 菜单名

### 5.2 接入前必须完成的动作

前后端联调前必须先补齐《XSS富文本字段协议清单》，至少确认：

- 应用/模块
- 接口路径
- 请求字段路径
- 是否允许富文本
- 后端采用配置放行还是字段注解
- 前端组件约束要求
- 允许的标签/属性/协议范围

没有进入协议清单的字段，一律按普通文本处理。

### 5.3 当前项目已经确认的字段边界

当前 `sso-auth` 已经提前确认了一批字段边界，前端可以直接按协议清单执行：

- `appName`、`roleName`、`deptName`、`postName`、`dictName`、`dictLabel`、`username`、`nickname` 都是普通文本
- `remark`、`description` 一类字段在当前项目里也按普通文本处理，不能因为“是备注”就直接渲染 HTML
- `comPath`、`path`、`avatar` 这类字段分别属于路径或 URL 字段，不属于富文本
- `password` 属于敏感字段例外，只能走密码框，不参与富文本协议
- `configValue` 当前也按普通文本处理，不能把它直接当成 HTML 模板字段使用

如果前端后续要做公告正文、帮助中心正文、模板正文这类富文本页面，必须先和后端一起更新《XSS富文本字段协议清单》，再开始联调。

### 5.4 前端富文本组件要求

当前仓库没有固定某一个富文本编辑器，因此这次不绑定具体组件库。
但无论用什么组件，都必须满足：

- 禁止脚本标签
- 禁止内联事件属性
- 禁止危险协议，例如 `javascript:`、`vbscript:`、`data:`
- 仅放开业务需要的标签和属性
- 上传图片、链接、表格等能力按需开启，不默认全开

## 6. multipart 场景注意点

这是本轮新增的重点：

- 后端已经支持 `multipart/form-data` 文本字段和文本分片清洗
- 但这不代表前端可以在文本分片里传任意 HTML
- 文件上传时，文本字段仍按普通文本或富文本协议处理
- 二进制文件内容不会做 XSS 文本清洗

因此前端在上传页面里要区分：

- 文件 part：按文件处理
- 文本 part：按普通文本或富文本协议处理

## 7. 联调与排查

### 7.1 如果前端发现内容被清洗

请按下面顺序排查：

1. 先确认字段是不是普通文本字段
2. 再确认页面有没有错误使用 `v-html` 或等价危险渲染
3. 确认提交内容里是否真的带了 HTML、事件属性、危险协议
4. 对照《XSS富文本字段协议清单》确认该字段是否允许富文本
5. 与后端确认该字段当前走的是哪种策略：
   - `TEXT`
   - `SEARCH`
   - `RICH_TEXT`
   - `NONE`

### 7.2 如果要排查后端策略是否命中

后端已接入 Micrometer 审计指标，可通过运维或后端协助查看：

- `sso.xss.hit.total`
- `sso.xss.hit.whitelist`
- `sso.xss.request.hit`

排查时建议同时提供：

- 请求路径
- 字段名
- 原始提交内容
- 当前页面渲染方式
- 后端返回值

## 8. 前端需要改哪些公共能力

建议优先统一治理这些公共层：

- 富文本编辑器封装
- HTML 预览组件封装
- 通用文本展示组件
- 请求参数拼装工具
- DOM 插入类工具函数

建议至少完成：

1. 统一禁止普通文本展示走 `v-html`
2. 富文本展示统一收口到公共组件
3. 上传页面把文本 part 和文件 part 明确分层处理
4. 历史页面排查所有直接 `innerHTML` 拼接逻辑
