# 数据库运行时配置改造说明

## 改造目标

本次将以下原本写在 `application.yaml` 中的关键配置迁移为从 `t_sys_config` 读取：

- `default.password`
- `sms-chinese.uid`
- `sms-chinese.key`
- `sms-chinese.send-url`
- `sms.aliyun.access-key-id`
- `sms.aliyun.access-key-secret`
- `sms.aliyun.endpoint`
- `sms.aliyun.region`
- `sms.aliyun.sign-name`
- `security.argon2.pepper`

## 现状分析结论

### 旧读取方式

- 默认密码原先通过 `@Value("${default.password:123456}")` 注入，使用点在：
  - `sso-auth/src/main/java/com/zhanglx/sso/auth/service/impl/AuthServiceImpl.java`
  - `sso-auth/src/main/java/com/zhanglx/sso/auth/service/impl/UserServiceImpl.java`
- Argon2 Pepper 原先通过 `@Value("${security.argon2.pepper:}")` 注入，使用点在：
  - `sso-auth/src/main/java/com/zhanglx/sso/auth/config/Argon2PasswordEncoder.java`
- 短信渠道配置原先通过 `@ConfigurationProperties(prefix = "sso.sms")` 绑定，使用点在：
  - `sso-sms/src/main/java/com/zhanglx/sso/sms/properties/SmsProperties.java`
  - `sso-sms/src/main/java/com/zhanglx/sso/sms/service/provider/SmsChineseChannelSender.java`
  - `sso-sms/src/main/java/com/zhanglx/sso/sms/service/provider/AliyunSmsChannelSender.java`

### 可复用能力

- 项目已存在 `t_sys_config`、`ConfigPO`、`ConfigMapper`、`ConfigServiceImpl` 和参数管理页面。
- 项目已存在 Redis 缓存体系，但本次没有把高敏配置放进 Redis，而是采用本地内存缓存，降低密钥暴露面。

### 敏感配置识别

- 高敏：
  - `default.password`
  - `sms-chinese.key`
  - `sms.aliyun.access-key-secret`
  - `security.argon2.pepper`
- 普通：
  - `sms-chinese.uid`
  - `sms-chinese.send-url`
  - `sms.aliyun.access-key-id`
  - `sms.aliyun.endpoint`
  - `sms.aliyun.region`
  - `sms.aliyun.sign-name`

## 最终架构

### 统一配置访问层

- 核心接口：`sso-core/src/main/java/com/zhanglx/sso/core/config/runtime/SystemConfigProvider.java`
- 统一键常量：`sso-core/src/main/java/com/zhanglx/sso/core/config/runtime/SystemConfigKeys.java`
- 数据库实现：`sso-auth/src/main/java/com/zhanglx/sso/auth/service/runtime/DatabaseSystemConfigProvider.java`

### 业务侧封装

- 默认密码与 Pepper 由 `AuthSecurityConfigService` 统一提供。
- 短信渠道配置由 `SmsChannelRuntimeConfigResolver` 统一组装。
- 业务代码不再自己查 `ConfigMapper`，也不再直接读这批 yml 配置。

## 表结构设计

复用现有 `t_sys_config`，补充以下字段：

- `config_group`：参数分组
- `sensitive_flag`：敏感标识
- `status`：启停状态

保留原有字段：

- `id`
- `config_name`
- `config_key`
- `config_value`
- `config_type`
- `remark`
- `create_time`
- `update_time`

## 缓存策略

- 实现方式：`DatabaseSystemConfigProvider` 使用本地 `ConcurrentHashMap` 做缓存。
- 读取策略：首次读取加载，之后命中本地缓存。
- 过期策略：默认 300 秒，可通过 `sso.config.runtime.cache-ttl-seconds` 调整。
- 刷新策略：
  - 参数新增、修改、删除后，后端自动刷新对应缓存。
  - 管理台新增了“刷新运行时缓存”接口，可手动清空本地缓存。
- 多节点说明：
  - 当前方案属于“本地缓存 + TTL 最终一致性”。
  - 同一节点上的管理台修改会立即生效。
  - 其他节点会在缓存过期后自动收敛，后续如需更强一致性，可扩展为 MQ / Redis PubSub 广播刷新。

## 启动顺序与安全考虑

- Argon2 Pepper 在 `Argon2PasswordEncoder` 初始化时通过 `AuthSecurityConfigService` 从数据库读取。
- 之所以不做 Pepper 热更新，是因为 Pepper 变更会影响既有密码校验结果，属于需要计划发布的高风险操作。
- 阿里云短信客户端会根据数据库配置指纹自动重建，避免凭证变更后仍使用旧客户端。

## 敏感配置治理

- 管理台读取敏感配置时统一返回 `******`，不再把真实值回显到前端。
- 参数新增、修改接口关闭了请求体和响应体的操作日志采集，避免日志落盘明文。
- 运行时配置异常只输出配置键，不输出敏感值本身。

## 验收建议

### 自动化测试

- `DatabaseSystemConfigProviderTest`
- `ConfigValueMaskingSupportTest`
- `SmsChannelRuntimeConfigResolverTest`

### 人工验证

1. 执行 `docs/sql/运行时配置升级脚本.sql` 或使用更新后的初始化 SQL 初始化数据库。
2. 启动 `sso-auth`。
3. 在参数管理页面确认这 10 个配置项存在，敏感项显示为 `******`。
4. 触发后台管理员重置密码，确认使用的是数据库中的 `default.password`。
5. 修改 `security.argon2.pepper` 后重启应用，确认密码组件能正常初始化。
6. 修改 `sms-chinese` 或 `sms.aliyun` 凭证后，通过“刷新运行时缓存”按钮触发缓存刷新，再验证短信发送。
7. 检查应用日志，确认没有明文输出短信密钥、阿里云 Secret 和 Pepper。

## 风险与边界

- 现网数据库必须先执行升级脚本，否则新增字段缺失会导致参数管理和运行时读取失败。
- Pepper 改动不建议在线热切换，推荐走发布窗口。
- 当前跨节点缓存刷新是 TTL 最终一致性，不是强一致广播。
