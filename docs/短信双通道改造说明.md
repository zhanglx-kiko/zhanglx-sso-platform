# 短信双通道改造说明

## 1. 当前短信体系现状分析

### 1.1 原短信能力现状
- 原实现位于 `sso-auth` 模块，验证码服务、短信通调用封装、短信通配置都耦合在认证服务内部。
- 原发送入口是 `MemberVerificationCodeServiceImpl`，调用方为 `AuthMemberController -> MemberAuthServiceImpl -> MemberVerificationCodeServiceImpl`。
- 原第三方实现只有短信通，且仅支持验证码短信文案拼装，不支持统一场景模板体系。
- 原配置散落在 `sso.sms.verification-code.scene-templates` 与 `sso.sms.sms-chinese`，渠道抽象不足，后续扩展新厂商成本高。

### 1.2 当前验证码链路现状
- 验证码主链路仍保持 `生成验证码 -> 写入 Redis -> 发送短信 -> 校验验证码`。
- Redis Key 规则保留：
  - `sso:member:verification:code:`
  - `sso:member:verification:resend:`
  - `sso:member:verification:phone-window:`
  - `sso:member:verification:ip-daily:`
- 验证码位数固定为 6 位。
- Redis 中验证码有效期固定为 180 秒。
- 频控逻辑保留：重发间隔、手机号 30 分钟窗口次数、单 IP 日限额。
- 校验逻辑保留：只校验 Redis 中已经存在的验证码，校验成功后删除。

### 1.3 当前短信类型支持现状
- 现有业务已经使用的场景：注册、忘记密码、绑定新手机号。
- 本次统一扩展后支持 5 个短信场景：
  - `REGISTER`
  - `CHANGE_BOUND_PHONE`
  - `FORGOT_PASSWORD`
  - `BIND_PHONE`
  - `VERIFY_BIND_PHONE`
- 业务层只传场景枚举，不再直接拼模板号或短信文案。

### 1.4 本次切入点
- 将短信发送能力整体迁移到 `sso-sms` 模块。
- `sso-auth` 只保留验证码业务规则与 Redis 链路，不再关心厂商 SDK 细节。
- 通过统一网关、统一请求、统一结果和配置切换实现双通道能力。

## 2. 最终双通道架构说明

### 2.1 统一抽象
- `SmsGateway`：统一短信发送入口。
- `SmsSendRequest`：统一请求对象，承载场景、手机号、模板参数、业务流水号。
- `SmsSendResult`：统一结果对象，承载渠道类型、成功标记、第三方状态码、消息、RequestId、BizId、OutId 等信息。
- `SmsChannelSender`：厂商渠道策略接口。

### 2.2 渠道实现
- `SmsChineseChannelSender`：保留原短信通能力，并增强为支持统一场景模板。
- `AliyunSmsChannelSender`：新增阿里云号码认证短信能力。
- `DefaultSmsGateway`：根据 `sso.sms.provider` 选择具体渠道。

### 2.3 调用链
- `AuthMemberController`
- `MemberAuthServiceImpl`
- `MemberVerificationCodeServiceImpl`
- `SmsGateway`
- `SmsChineseChannelSender / AliyunSmsChannelSender`

## 3. 阿里云短信接入说明

### 3.1 固定参数
- `signName` 固定使用 `速通互联验证平台`
- `returnVerifyCode` 固定使用 `false`
- `validTime` 固定使用 `180`
- `templateParam` 固定传 `code` 与 `min`
  - `code` 来自 Redis 已存验证码
  - `min` 固定传 `3`

### 3.2 成功判定
必须同时满足以下条件才视为成功：
- HTTP 状态码为 200
- `Success = true`
- `Code = OK`
- `Message` 为空或包含“成功”

### 3.3 返回映射
`AliyunSmsVerifyCodeResponsePayload` 对阿里云响应做了完整映射，保留：
- `AccessDeniedDetail`
- `Message`
- `RequestId`
- `Code`
- `Success`
- `Model.VerifyCode`
- `Model.RequestId`
- `Model.OutId`
- `Model.BizId`

## 4. 现有短信方案增强说明
- 短信通渠道改为使用统一 `SmsSendRequest` 入参。
- 短信通文案从配置化模板映射中渲染，不再在业务服务内拼接文案。
- 短信通返回统一转换为 `SmsSendResult`。
- 短信通错误码通过统一翻译器转换为稳定的业务异常。

## 5. 配置切换说明
- 当前启用渠道：`sso.sms.provider`
- 可选值：
  - `sms-chinese`
  - `current`
  - `aliyun`
- 切换渠道时无需修改业务代码，只需修改配置。

### 5.1 核心配置示例
```yaml
sso:
  sms:
    provider: sms-chinese
    verification-code:
      mock-send-enabled: false
    sms-chinese:
      enabled: true
      uid: ${SMSCHINESE_UID:}
      key: ${SMSCHINESE_KEY:}
    aliyun:
      enabled: true
      access-key-id: ${ALIYUN_SMS_ACCESS_KEY_ID:}
      access-key-secret: ${ALIYUN_SMS_ACCESS_KEY_SECRET:}
```

## 6. 模板场景与 templateCode 映射
| 场景 | templateCode | 说明 |
| --- | --- | --- |
| `REGISTER` | `100001` | 登录/注册 |
| `CHANGE_BOUND_PHONE` | `100002` | 修改绑定手机号 |
| `FORGOT_PASSWORD` | `100003` | 重置密码 |
| `BIND_PHONE` | `100004` | 绑定新手机号 |
| `VERIFY_BIND_PHONE` | `100005` | 验证绑定手机号 |

## 7. Redis 验证码链路兼容说明
- 验证码固定 6 位。
- Redis 中验证码有效期固定 180 秒。
- 模板参数 `code` 直接复用 Redis 已存验证码，不重新生成第二份验证码。
- 模板参数 `min` 固定传 3。
- 重发间隔、手机号窗口限流、单 IP 日限流、校验成功删除验证码等逻辑均保留。

## 8. 阿里云 RAM 授权要求
最小权限如下：
- Action：`dypns:SendSmsVerifyCode`
- Resource：`*`
- Condition：无需产品级 Condition
- RAM 产品代码：`dypns`

## 9. 测试说明

### 9.1 已补充的自动化测试
- `SmsSceneTypeTest`：验证场景解析与别名兼容。
- `SmsTemplateSupportTest`：验证模板渲染与模板号映射。
- `DefaultSmsGatewayTest`：验证渠道切换与阿里云错误码翻译。

### 9.2 建议人工验收步骤
1. 配置 `sso.sms.provider=sms-chinese`，发送 `REGISTER` 场景验证码，确认短信内容模板正确。
2. 配置 `sso.sms.provider=aliyun`，发送 `REGISTER` 场景验证码，确认阿里云返回 `Code=OK`、`Success=true`。
3. 将 `mock-send-enabled=true`，再次发送验证码，确认 Redis 中存在验证码，且日志仅在模拟发送模式下输出验证码。
4. 检查 Redis 中 `sso:member:verification:code:*` 的值，确认与短信实际发送的 `code` 一致。
5. 检查验证码长度始终为 6 位。
6. 检查阿里云请求参数，确认 `signName=速通互联验证平台`、`returnVerifyCode=false`、`validTime=180`。
7. 验证 `CHANGE_BOUND_PHONE`、`FORGOT_PASSWORD`、`BIND_PHONE`、`VERIFY_BIND_PHONE` 的模板编号与文案映射是否正确。
8. 构造阿里云错误码 `MOBILE_NUMBER_ILLEGAL`、`BUSINESS_LIMIT_CONTROL`、`FREQUENCY_FAIL`、`INVALID_PARAMETERS`、`FUNCTION_NOT_OPENED`，确认上层错误信息稳定、日志保留关键排障字段。

## 10. 风险与边界说明
- 当前短信通依赖仍以本地 `systemPath` 方式引入，Maven 会继续给出 warning，但编译可通过。
- `sso-auth` 资源复制阶段受本机占用影响时，仍建议使用 `-Dmaven.resources.skip=true` 做源码编译验证。
- 项目内仍有少量与短信无关的历史乱码文件，本次只修复短信相关链路与直接触达的认证接口文件。

## 11. 后续扩展建议
- 将短信通本地 Jar 安装进私有仓库或本地 Maven 仓库，移除 `systemPath` warning。
- 如后续增加更多短信厂商，只需新增 `SmsChannelSender` 实现，并补充配置与错误码映射。
- 如果后续要支持通知类短信，可在当前 `SmsSceneType` 基础上继续扩展，不需要改调用层结构。
