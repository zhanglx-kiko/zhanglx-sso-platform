# 外部平台接入操作日志说明

## 1. 功能说明

统一操作日志能力用于记录后台管理、任务调度、消息消费、开放接口等场景中的审计日志。

日志主存储为 Elasticsearch，支持：

- 高并发异步写入
- 多维过滤
- 模糊检索
- 审计详情查看
- traceId/requestId 贯穿

## 2. 架构说明

推荐接入链路：

`业务代码 -> @OperationLog 或 OperationLogClient -> 摘要/脱敏 -> 有界队列 -> 独立日志线程池 -> ES Bulk`

不允许在业务线程中同步逐条写 ES。

## 3. 为什么操作日志存 ES

- 量大、写多、查多、检索需求强
- MySQL 不适合作为操作日志主存储
- ES 更适合日志型场景的全文检索与组合过滤

## 4. 接入方式

### 4.1 注解方式

适合 Web Controller 或明确的服务入口。

```java
@OperationLog(
        module = "用户管理",
        feature = "用户",
        operationType = "CREATE",
        operationName = "新增用户",
        operationDesc = "新增后台系统用户",
        includeResponseBody = false
)
@PostMapping("/users")
public void create(@RequestBody @Valid UserDTO dto) {
    userService.addUser(dto);
}
```

### 4.2 手动 API 方式

适合定时任务、消息消费、非 Web 场景、批处理。

```java
@RequiredArgsConstructor
@Service
public class DemoJobService {

    private final OperationLogClient operationLogClient;

    public void executeJob(String batchNo) {
        operationLogClient.record(OperationLogCommand.builder()
                .module("任务中心")
                .feature("批处理任务")
                .operationType("EXECUTE")
                .operationName("执行批处理任务")
                .operationDesc("执行每日账务汇总任务")
                .requestPath("/job/accounting/daily")
                .resultStatus("SUCCESS")
                .startTime(LocalDateTime.now())
                .endTime(LocalDateTime.now())
                .durationMs(120L)
                .ext(Map.of("batchNo", batchNo))
                .build());
    }
}
```

## 5. 字段说明

核心字段如下：

- `logId`：日志ID
- `appCode/appName`：应用标识
- `platformCode/platformName`：平台标识
- `module/feature`：模块与功能
- `operationType/operationName/operationDesc`：操作类型、名称、描述
- `userId/username/displayName`：操作人
- `requestMethod/requestPath/requestQuery/requestBodySummary`
- `responseSummary`
- `resultStatus/errorCode/errorMessageSummary`
- `exceptionType/exceptionStackSummary`
- `clientIp/userAgent`
- `traceId/requestId`
- `startTime/endTime/durationMs`
- `sourceSystem`
- `ext`
- `ingestTime`

## 6. 脱敏与截断说明

系统不会原样无上限记录请求体、响应体、异常堆栈。

统一规则：

- 敏感字段自动脱敏
- 大对象只保留摘要
- 集合/Map/Bean 预览数量有限制
- 超长字段统一截断并附带 `...(truncated)`

典型敏感字段：

- `password`
- `token`
- `secret`
- `authorization`
- `phone`
- `idCard`

## 7. traceId 说明

traceId/requestId 默认由网关与服务端 Filter 自动补齐，并通过 Feign 自动透传。

如果是非 Web 场景，建议接入方主动补充：

```java
OperationLogCommand.builder()
        .traceId("trace-demo-001")
        .requestId("request-demo-001")
        .build();
```

## 8. 错误处理说明

日志链路异常不会影响主业务成功与否。

当前处理策略：

- 队列满：按策略丢弃当前或最旧日志
- ES 不可用：有限次重试，超过次数直接丢弃
- Bulk 部分失败：只重试失败部分
- 摘要序列化失败：降级为字符串

## 9. 最佳实践

- Controller 写操作优先使用 `@OperationLog`
- 非 Web 场景统一注入 `OperationLogClient`
- `module/feature/operationType` 尽量固定枚举化，方便跨平台统一检索
- `ext` 只放少量、稳定、可控的扩展字段
- 时间范围查询必须带合理区间，避免一次扫过多天索引

## 10. 常见问题

### 10.1 为什么看不到完整请求体？

因为日志系统默认只记录脱敏后的摘要，防止敏感数据泄露和大字段撑爆写入链路。

### 10.2 为什么分页超过一定深度会报错？

因为 ES 深分页成本很高。超过 `searchMaxWindow` 后需要改用 `searchAfterToken`。

### 10.3 为什么不把操作日志顺便存 MySQL 一份？

因为双写 MySQL 会显著增加主库压力，也会让日志系统重新变成故障源。

## 11. 注意事项

- 不要把密码、Token、Secret 直接塞进 `ext`
- 不要把原始大响应对象直接当作手动日志参数
- 不要把日志写入失败当作业务失败条件
- 不要把所有平台随意写入动态字段

## 12. ES 索引说明

- 索引前缀：`operation-log`
- 按天滚动：`operation-log-2026-04-08`
- 模板：`sso-log/src/main/resources/es/operation-log-index-template.json`
- `dynamic: false`
- `ext: flattened`

## 13. 查询说明

后台查询接口：

- `POST /apis/v1/auth/s/operation-logs/page`
- `GET /apis/v1/auth/s/operation-logs/{logId}`

查询规则：

- 精确匹配使用 `term`
- 时间范围使用 `range`
- 模糊检索使用 `multi_match`

## 14. 建议配置

建议从以下配置开始：

```yaml
sso:
  log:
    operation:
      enable-operation-log: true
      elasticsearch:
        hosts:
          - 127.0.0.1:9200
        index-prefix: operation-log
        bulk-batch-size: 200
        bulk-flush-interval-ms: 1000
        bulk-max-bytes: 1048576
        max-retry-count: 1
      async:
        core-pool-size: 1
        max-pool-size: 2
        queue-capacity: 5000
        rejection-policy: DROP_CURRENT
```
